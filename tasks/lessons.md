# 开发经验教训

## 2026-03-12: ETF AI系统初始开发

### 架构设计经验

1. **模块化设计的重要性**
   - 将系统分为5层（数据层、策略层、Agent层、执行层、存储层）使得职责清晰
   - 每个模块独立可测试，降低耦合度

2. **配置驱动的优势**
   - 使用YAML配置文件 + Pydantic验证，既可读又类型安全
   - 策略参数、ETF池、Agent配置分离，便于管理和版本控制

3. **数据标准化的必要性**
   - akshare返回的数据格式需要统一标准化
   - 建立内部标准格式（trade_date, open, close等）避免后续混乱

### 技术选型经验

1. **依赖库选择**
   - akshare: 免费、易用，适合A股/ETF数据获取
   - SQLAlchemy 2.0: 功能强大，类型提示完善
   - Pydantic: 配置验证和数据模型的最佳选择
   - loguru: 比标准logging更优雅

2. **避免过度依赖**
   - 回测引擎选择自实现而非backtrader，保证与生产逻辑一致
   - 简单场景不需要复杂框架

### 开发流程经验

1. **先核心后外围**
   - 优先实现策略引擎核心逻辑
   - Agent层可以后续逐步完善
   - 这样可以更快验证策略可行性

2. **数据验证不可少**
   - 在策略计算前必须验证数据质量
   - 缺失值、异常跳变都会影响策略结果

3. **日志记录要充分**
   - 每个关键步骤都要记录日志
   - 便于调试和回溯问题

### 待改进事项

1. **测试覆盖**
   - 当前缺少单元测试
   - 需要为每个模块编写测试用例

2. **错误处理**
   - 部分模块的异常处理可以更完善
   - 需要统一错误处理策略

3. **性能优化**
   - 数据获取可以增加缓存机制
   - 批量操作可以并行化

4. **文档完善**
   - 需要为每个模块编写详细的API文档
   - 增加使用示例和最佳实践

### 下次开发注意事项

1. 完成存储层时，注意数据库事务管理
2. 实现回测引擎时，确保与实盘逻辑完全一致
3. Agent层实现时，注意LLM调用的成本控制和错误处理
4. 执行层实现时，风控规则要严格，宁可保守不可激进

## 2026-03-24: Task 3 review 更正

1. **成交价不能偷用历史价格**
   - 执行/回测统一语义时，成交必须严格依赖 `trade_date` 当日价格。
   - `get_latest_price_on_or_before` 只适合研究查询，不适合成交撮合或当日估值。

2. **预检查与真执行必须共享同一状态来源**
   - 如果执行阶段从 repo 读最新组合状态，检查阶段也必须用同一解析规则。
   - 请求里的 `current_position/current_holding_shares` 只能作为 repo 缺失时的回退，不能与执行阶段各走一套。
